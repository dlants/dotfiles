# context

The goal is to replace the existing `vim-grepper` plugin with a new Lua plugin: **shuck**. It is a generic "edit a shell command, run it, browse the results" tool — not specific to grep, though `rg ... --vimgrep` is the default command.

- Opens a floating prompt at the top of the screen, **pre-populated** with `rg -H --no-heading --vimgrep ` (no auto-execute).
- Streams the command's stdout into an attached results buffer below the prompt.
- Lets the user navigate the results, open a single result, or push the whole result set to the quickfix list.

"Open this result" and "send to quickfix" both assume vimgrep-style lines (`path:lnum:col:text`). Other commands work fine — they just won't have an openable result, and `setqflist` will skip lines that don't match its `errorformat`.

## Relevant files and entities

- `nvim/lua/needle.lua` — reference implementation. Patterns to copy:
  - Floating-window pair (prompt window + results window) created in `open_windows`.
  - Picker state table (`picker` upvalue) holding buffers, windows, timers, and child handle.
  - `start_discovery` using `vim.system` with a streaming `stdout` callback and a `kill` closure returned for cancellation.
  - Selection rendering via extmarks in `NS_HL` and `render_results`.
  - Keymap setup pattern in `setup_keymaps`.
  - Lifecycle helpers `close_picker`, `M.open`, `M.toggle`, `M.setup`.
- `nvim/lua/config/plugins.lua` — currently configures `vim.g.grepper` and the `<leader>g` mapping; will be updated to call the new plugin and remove vim-grepper config.
- `nvim/lua/config/pack.lua` — lists `mhinz/vim-grepper`; will be removed once the new plugin is in place.

## Key types/state

```lua
-- single global picker state, similar to needle's `picker`
state = {
  -- windows / buffers
  prompt_buf, prompt_win,    -- single-line prompt with the editable command
  results_buf, results_win,  -- buffer that displays raw stdout

  -- command + execution
  cmd_string,                -- text of the prompt buffer, last submitted
  child,                     -- vim.system handle for the running process

  -- streamed output (in memory; the buffer is just a viewport)
  output_lines,              -- table<int, string>, all complete output lines so far
  pending_chunk,             -- string, partial trailing line not yet split on \n
  rendered_count,            -- number of lines currently in results_buf

  -- selection
  selected,                  -- 1-based index into output_lines

  -- status
  running,                   -- true while a child process is alive

  -- bookkeeping
  flush_timer,               -- coalesces streamed chunks into a single redraw
  spinner_timer,             -- ticks the title-bar spinner while running
  cwd,                       -- working dir for the command
  stale,                     -- true after the prompt is edited, false right after a run
}
```

## Output handling

Output is **streamed and stored in memory**, with the results buffer acting as a viewport over the first `max_render` (~200) lines. This keeps redraw work bounded regardless of how much output the command produces, and lets us push every result to the quickfix list — not just the visible ones.

State is updated in two layers:

- `state.output_lines` (Lua table) — every complete line from stdout, up to `state.max_results` (e.g. 10000). The source of truth for "open selected" and "send to quickfix".
- `results_buf` — a thin viewport. Only the first `max_render` rows are written via `nvim_buf_set_lines`. Rendering is **debounced**: streamed chunks accumulate in `output_lines` immediately, but the buffer is only rewritten on a timer tick (`stream_flush_ms` ~30ms).

Parsing is lazy:

- **Open selected**: parse `state.output_lines[state.selected]` with `^([^:]+):(%d+):(%d+):(.*)$`. If it doesn't match, no-op.
- **Send to quickfix**: hand `state.output_lines` directly to `vim.fn.setqflist({}, ' ', { lines = output_lines, efm = '%f:%l:%c:%m', title = state.cmd_string })`. Vim parses; non-matching lines are silently dropped.

### Avoiding overwhelming the editor

Streaming naïvely can flood the main thread with `vim.schedule` callbacks and per-chunk `nvim_buf_set_lines` calls. Mitigations baked into the design:

1. **Single coalesced redraw** — `on_stdout` only appends to `output_lines` and arms `flush_timer`; it never touches the buffer or scrollback directly. The timer fires at most once per `stream_flush_ms`, regardless of chunk frequency.
2. **Bounded buffer writes** — the viewport never grows past `max_render` rows. Once it's full, redraws are no-ops on the buffer side; only `output_lines` keeps appending (cheap `t[#t+1] = line`).
3. **Hard memory cap** — when `#output_lines >= max_results`, the child is killed (`state.child:kill("sigterm")`) and the title gains a `(truncated)` suffix. Default 10000 lines (~1MB at 100 char/line) is plenty for any sane interactive use. The buffer itself stays clean — no inline truncation marker.
4. **Single-line splitting in Lua** — chunk handling is pure-string ops in a `vim.schedule` block; no nvim API call inside the hot loop.
5. **Stop on consumer events** — opening a result, sending to quickfix, or closing the picker all kill the child immediately so a long-running command can't keep streaming into a dead picker.

## Keybinding scheme (in the prompt buffer)

`<CR>` always opens the selected result; `<S-CR>` always runs (or re-runs) the command. Insert-mode `<CR>` runs as well, since at that point the user is editing the command and "commit" is the natural action.

| key             | mode   | action                                                |
| --------------- | ------ | ----------------------------------------------------- |
| `<CR>`          | insert | run the command (commit edit)                         |
| `<CR>`          | normal | open the currently-selected result                    |
| `<S-CR>`        | both   | run (or re-run) the command                           |
| `<C-j>`/`<Down>`| both   | move selection down                                   |
| `<C-k>`/`<Up>`  | both   | move selection up                                     |
| `<C-x>`         | both   | open selected result in horizontal split              |
| `<C-v>`         | both   | open selected result in vertical split                |
| `<C-t>`         | both   | open selected result in new tab                       |
| `<C-q>`         | both   | send all parsed results to quickfix and close         |
| `<Esc>`         | insert | leave insert (default vim)                            |
| `<Esc>`         | normal | close the picker                                      |
| `<C-c>`         | both   | close the picker                                      |

Rationale: in normal mode the user is "browsing" results, so `<CR>` should match every other browser/picker convention and open the highlighted entry. `<S-CR>` is the explicit "run" verb that works regardless of mode.

`<S-CR>` requires a terminal that distinguishes it from plain `<CR>` (kitty keyboard protocol, ghostty, modern wezterm, neovide, GUI nvim). The user's setup (ghostty) handles this.

  - `M.config.default_cmd = "rg -H --no-heading --vimgrep "` (trailing space) and `max_render` for the results buffer.
  - `M.setup` registers `:Shuck` user-command and default highlight groups (`ShuckSel`, `ShuckMatch` linked sensibly).
  - Testing
    - Behavior: requiring the module and calling `setup({})` does not error and exposes `:Shuck`.
    - Setup: fresh nvim, `:lua require('shuck').setup({})`.
    - Actions: run `:command Shuck`.
    - Expected output: command is registered.
    - Assertions: `vim.fn.exists(':Shuck') == 2`.

- [ ] **Floating windows** — port `open_windows` from needle but tune sizing.
  - Top of screen, full width minus 2 cols.
  - Prompt window: 1 line tall with rounded border, title `" shuck "`.
  - Results window: ~`min(20, floor(0.5 * lines))` tall, directly below prompt, non-focusable, `wrap=false`.
  - Both buffers `nofile` / `bufhidden=wipe` / `filetype=shuck`.
  - On open, write `M.config.default_cmd` into the prompt buffer and `startinsert!` with cursor at end of line.
  - Testing
    - Behavior: opening shows the prompt populated and cursor at end.
    - Setup: `M.open({})`.
    - Actions: read prompt buffer line 1; query window cursor.
    - Expected output: line equals default cmd; cursor column equals `#default_cmd + 1`.
    - Assertions: `assert.equals` on each.

- [ ] **Command execution** — `run_command()`:
  - Read prompt buffer line 1 → `state.cmd_string`.
  - If `state.child` is alive, `state.child:kill("sigterm")` first; null it out.
  - Reset `state.output_lines = {}`, `state.pending_chunk = ""`, `state.rendered_count = 0`, `state.selected = 1`, `state.running = true`. Clear the results buffer (one `nvim_buf_set_lines` call with `{}`).
  - Start the spinner: arm `spinner_timer` to tick every ~80ms updating the title.
  - Spawn `vim.system({ "sh", "-c", state.cmd_string }, { cwd = state.cwd, text = true, stdout = on_stdout, stderr = on_stderr }, on_exit)`.
  - `on_stdout` (libuv thread → wrap in `vim.schedule`):
    - Concatenate chunk onto `state.pending_chunk`, find the last `\n`, split everything up to it into lines, append each to `state.output_lines` (skipping empty lines), keep the trailing fragment in `state.pending_chunk`.
    - If `#state.output_lines >= max_results`: `state.child:kill("sigterm")`.
    - Arm `flush_timer` for `stream_flush_ms` (idempotent: just `:stop()` then `:start()`).
  - `on_stderr` (libuv thread → wrap in `vim.schedule`):
    - Same chunk-handling as `on_stdout`, but each emitted line is prefixed with `!! ` before being appended to `state.output_lines`. The prefix won't match the vimgrep `efm`, so stderr lines are visible but silently skipped by `to_quickfix` and `open_selected`.
  - `on_exit`:
    - Flush any remaining `pending_chunk` and stderr fragment as final lines.
    - Set `state.running = false`, stop spinner, do one final render.
  - Testing
    - Behavior: streaming works end-to-end with bounded redraws.
    - Setup: open with cmd `seq 1 5000 | sed 's|^|f:|; s|$|:1:x|'`, set `max_render = 50`, `max_results = 1000`.
    - Actions: trigger run; wait for `state.running == false`.
    - Expected output: `#state.output_lines == 1000` (capped); `nvim_buf_line_count(results_buf) == 50`; child terminated by signal.
    - Assertions: count fields plus presence of "(truncated)" marker.

- [ ] **Rendering & selection** — port `render_results` with these changes:
  - No fzy match positions (we're not filtering).
  - Slice `state.output_lines[1..max_render]` into `results_buf` via a single `nvim_buf_set_lines` call. Update `state.rendered_count` to that slice length.
  - Selected row highlighted with `ShuckSel` extmark on the buffer row equal to `state.selected` (only if `selected <= rendered_count`; otherwise no highlight).
  - `move_selection(delta)` wraps modulo `#state.output_lines`, redraws highlight, and scrolls the results window so the selected row is visible.
  - Update prompt window title to reflect status. Spinner uses the braille set `⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏` rotated by `spinner_timer`:
    - while running: ` shuck ⠋ N lines `
    - after exit:    ` shuck • N lines ` (or ` shuck • N lines (truncated) ` if the memory cap was hit)
  - The spinner timer (`spinner_timer`) only runs while `state.running == true`; on `on_exit` we stop it and do a final title write.
  - Testing
    - Behavior: `<C-j>` advances selection and visible highlight.
    - Setup: open + run with 5 parsed entries.
    - Actions: simulate `<C-j>` twice.
    - Expected output: `state.selected == 3`; extmark on row 3.
    - Assertions: read state and extmarks.

- [ ] **Open selected result** — `open_selected(open_cmd)`:
  - Read `state.output_lines[state.selected]`; match `^([^:]+):(%d+):(%d+):(.*)$`. If it doesn't match, no-op.
  - Kill any in-flight child, then close the picker.
  - `vim.cmd((open_cmd or "edit") .. " " .. fnameescape(file))`, then `nvim_win_set_cursor(0, { lnum, col-1 })`, then `normal! zz`.

- [ ] **Cancellation & cleanup** — `close_picker` and explicit kill points:

  - Call `kill_child()` at every consumer event:
    - on a fresh `run_command()`
    - on `open_selected()` (we're done with the picker)
